Raspberry Pi Pi-hole Network Lab

I set up Pi-hole on a Raspberry Pi to learn how DNS, DHCP, IPv4, IPv6, and router control work on a real home network. The goal was network-wide ad blocking. The useful part was figuring out why the Xfinity gateway kept bypassing it.

What I Debugged:

  • Raspberry Pi Server Setup: Installed Pi-hole headlessly, reserved 10.0.0.50, set Cloudflare as upstream DNS, and tested it with nslookup.
  • DNS Debugging: Queried Pi-hole directly, then checked the dashboard to prove requests were reaching the server.
  • DHCP vs DNS: Learned the difference between resolving names and assigning network settings, and why controlling DHCP is what lets you control where devices send DNS traffic.
  • Xfinity Router Limits: Found that the stock gateway would not reliably allow custom network-wide DNS, so devices kept falling back to Comcast DNS.
  • IPv6 Bypass: Tracked how Xfinity IPv6 DNS, including 2001:558:feed::1, could silently bypass Pi-hole even when IPv4 settings looked correct.
  • Risky Change Recovery: Tested a Pi-hole DHCP takeover, hit router/DHCP conflicts and temporary network loss, then backed out and isolated the problem instead of guessing.

This taught me more because it did not go perfectly. I learned to test one layer at a time, separate a working service from a router problem, and think before changing settings that affect the whole network.

Along the way, I got hands-on with:

  • Activation Functions: Tested ReLU, Sigmoid, and Tanh to see how they changed learning.
  • Preventing Overfitting: Tried regularization and dropout so the model would not only memorize the training data.
  • Matrix Math & Python Libraries: Used NumPy and vectorized operations to make the code faster and cleaner.

This project made machine learning feel less mysterious. It forced me to slow down, test each step, and understand why the model was or was not learning.
